#+title: Secrets Management
#+author: Pete 'Peteches' McCabe

* Overview

Secrets (API tokens, passwords, Tailscale auth keys, etc.) are encrypted with
[[https://getsops.io/][SOPS]] using [[https://age-encryption.org/][age]] public-key encryption.  Each VM has a unique age keypair; SOPS
encrypts secrets so only the intended recipient VM(s) can decrypt them at boot.

The =sops-guix= channel provides =sops-service-type=, which is wired into VMs via
the =sops-secrets= parameter of =make-vm-os= in =peteches/systems/vm-base.scm=.

* Directory structure

#+begin_example
.sops.yaml              ← encryption rules (path regex → age recipients)
age-keys/               ← age public keys, one per VM
secrets/
  hosts/
    <vm-name>/          ← per-host encrypted YAML files
  groups/
    concourse/          ← secrets shared by the Concourse cluster
    media/              ← secrets shared by media-stack VMs
  shared/
    tailscale.yaml      ← Tailscale auth key (shared)
#+end_example

* Age keys

Each VM generates its own age keypair on first boot (via the SOPS key generator
service in =vm-base.scm=).  The public key is committed to =age-keys/<vm-name>.pub=
so that new secrets can be encrypted for that VM without needing its private key.

The private key lives only on the VM itself (=/etc/age/key.txt= or similar);
it is never committed.

** Adding a key for a new VM

After the VM's first boot, retrieve and commit its public key:

#+begin_src bash
ssh <vm-name> cat /etc/age/key.txt | age-keygen -y   # prints public key
# DO NOT echo the private key — store only the public key:
# age-keys/<vm-name>.pub
#+end_src

Then add the appropriate =creation_rules= entry to =.sops.yaml=.

* Editing secrets

#+begin_src bash
sops secrets/hosts/<vm-name>/foo.yaml      # opens $EDITOR, re-encrypts on save
sops secrets/groups/media/credentials.yaml
#+end_src

SOPS reads encryption rules from =.sops.yaml= automatically.  The PGP key
=5FFB7006A36B0AEDE8D08E3753B1E002546A642A= (peteches operator key) is always
included as an additional recipient so secrets remain accessible from the
developer machine.

* Wiring secrets into a VM OS config

In the VM's system file, pass =sops-secrets= to =make-vm-os=:

#+begin_src scheme
(make-vm-os
  ...
  (sops-secrets (list "secrets/hosts/myvmname/foo.yaml")))
#+end_src

The =sops-service-type= will decrypt those files at activation time and make them
available to services on the VM.

* Group secrets

Some secrets are shared across multiple VMs.  Groups are defined in =.sops.yaml=
with a list of age recipients:

- =secrets/groups/concourse/= — shared by =concourse-web01= (other Concourse nodes
  get what they need via the web node)
- =secrets/groups/media/= — shared by jellyfin, prowlarr, arr, downloads, rustdesk

* Shared secrets

=secrets/shared/tailscale.yaml= contains the Tailscale auth key used by VMs that
run Tailscale.  It is encrypted for all Tailscale-enabled VM recipients.
